# Issue 駆動開発 (Issue-Driven Development) ワークフロー

このスキルは、本リポジトリにおける開発タスクの標準的なプロセスを定義します。要件の明確化、重要な段階でのユーザー承認、および Issue・ブランチ・プルリクエストの適切な紐付けを確実に行うためのものです。

## ワークフローの各ステップ

### 1. 要件分析と Issue ドラフト作成

- ユーザーの要件を分析し、要約します。
- GitHub Issue のたたき台を作成します（通常は `implementation_plan.md` アーティファクトとして作成）。
- **この段階ではまだコードに変更を加えてはいけません。**

### 2. ユーザーによるレビュー (ドラフト)

- 作成したドラフトをユーザーに提示し、レビューを依頼します。
- フィードバックに基づき、承認されるまで内容をブラッシュアップします。

### 3. Issue 作成とブランチ作成

- ドラフトが承認されたら以下の操作を行います：
  - GitHub 上に実際の Issue を作成します。
  - 作業用ブランチ（例: `fix/issue-<number>-<title>`）を作成します。
  - ブランチを Issue にリンクさせます。
  - 作成した作業用ブランチに移動します。
- 開発環境の準備（依存関係のインストール、ディレクトリのクリーンアップ等）を行います。

### 4. 変更開始の確認

- **必須**: 変更を開始する前に、ユーザーに変更を開始して良いか確認します。
- ユーザーからの承認を待ちます。

### 5. 実装フェーズ

- 計画に基づき変更を実装します。
- TDD（テスト駆動開発）やアーキテクチャのガイドラインに従います。

### 6. コミットメッセージのドラフトと完了報告

- 実装が完了したら以下の操作を行います：
  - 明確で分かりやすいコミットメッセージのたたき台を作成します。
  - ユーザーに完了報告を行い、メッセージのドラフトを提示します。
  - **この時点ではまだコミットやプッシュを行ってはいけません。変更はローカルのみに留めます。**

### 7. 反復的な修正と微調整

- ユーザーからの追加の指摘や指示に対応します。
- ユーザーが満足するまでこのプロセスを繰り返します。

### 8. 最終承認と PR 作成

- 変更内容とコミットメッセージが最終的に承認されたら：
  - ローカルで変更をコミットします。
  - GitHub 上でプルリクエスト (PR) を作成します。
  - PR を元の Issue に紐付けます（例: "Closes #22" を記述）。

## ベストプラクティス

- **チェックポイントをスキップしない**: 作業の開始前と変更の確定前には、必ずユーザーの承認を得てください。
- **追跡可能性 (Traceability)**: ブランチや PR は常に Issue 番号と紐付けてください。
- **アトミックなコミット**: タスクが大きい場合は、複数の PR に分割するかユーザーと相談してください。
